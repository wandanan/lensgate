# 边界情况分析

> 基于 Phase 2 实测发现的问题梳理 | 2026-06-28

---

## 1. 缓存命中但描述不全面  [ P0 ]

**场景**:
```
Turn 1: "描述这张图" → vision: "项目管理甘特图，深色主题，4/27~5/9..."
Turn 5: "右上角那个红色标签写的什么？" → 缓存命中，但描述里没提到红色标签
```

**根因**: 首次识图的 prompt 是通用描述，不针对用户后续追问的特定细节。

**影响**: 后续追问无法获得增量信息，对话质量退化。

**可行方案**:

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A. 关键词触发重识 | 检测用户消息中含 `图`/`颜色`/`左上`/`写什么` 等聚焦词时，重新 call vision 并附上用户追问 | 实现简单，解决大部分场景 | 关键词覆盖不全 |
| B. 始终带上下文重识 | 每次识图时把对话历史中与该图片相关的用户问题拼接进 vision prompt | 描述更精准 | token 开销大，可能重复描述无关信息 |
| C. 混合策略 | 首次用通用 prompt 缓存；追问问卷触发时用 "用户追问：xxx，请仔细描述图中此细节" 重新识别，结果更新缓存 | 兼顾效率与质量 | 需要缓存更新逻辑 |

**推荐**: **A** 先落地，成本最低。后续演进到 C。

---

## 2. 多张不同图片  [ P1 ]

**场景**: 一次对话贴了 3 张图（如截图对比），分别识图后需要正确编号对应。

**当前状态**: `recognize_batch` 已支持并行识图，rewriter 用 `[图片 n/total]` 标记。但因上文优化已改为 `[图片内容]\n<desc>` 格式，多图时需要在描述前加编号区分。

**待完善**:
- 多图时保持编号前缀 `[图片 1/3]\n<desc>` 以区分不同图
- 单图时省略编号，用简洁格式

---

## 3. Vision API 不可用  [ P1 ]

**场景**: Vision API key 过期/余额不足/服务宕机。

**当前状态**: 降级为 `[图片无法识别]`，目标模型收到该文本后无法理解图片内容。请求继续（不阻断），但对话质量崩。

**待完善**:
- 在降级文本中告知用户"视觉服务暂不可用"，而不仅是模型层面静默
- 可选：在 SSE 流中发送一个警告事件提示用户
- 监控告警：连续 N 次 vision 失败 → 日志 WARNING

---

## 4. 超大图片  [ P2 ]

**场景**: 50MB+ 的 TIFF/BMP 图片 base64 编码后请求体巨大。

**影响**:
- base64 解码 + vision 处理耗时长
- 可能突破 10MB Content-Length 限制（被 413 拦截）
- 内存占用高

**待完善**:
- 单张图片大小限制（如 20MB）
- 超限返回友好错误，提示用户压缩后再传
- 可选：客户端侧压缩/缩放后再发送

---

## 5. 流式场景用户等待  [ P2 ]

**场景**: 含图请求 + `stream: true`。vision 阶段耗时 8-30 秒，期间无任何 SSE 事件发出，用户看到长时间沉默。

**当前状态**: vision 完成后才开始流式转发目标模型 token。中间无进度通知。

**待完善**:
- vision 开始前发送一个 SSE 提示事件：`event: status data: {"type":"processing_image","message":"正在识别图片..."}`
- vision 完成后发送：`event: status data: {"type":"image_processed","duration_ms":8492}`

---

## 6. 图片 URL 源  [ P3 ]

**场景**: Anthropic 支持 `source.type: "url"` 和 `source.type: "base64"` 两种。

**问题**:
- URL 类型无法用 byte hash 缓存（URL 指向的内容可能变化）
- HTTP 下载可能超时/失败
- 部分 URL 需要认证或已过期

**待完善**:
- URL 类型用 `(url, etag/last-modified)` 做缓存 key
- 下载超时降级，不阻塞请求
- 实际场景中 Claude Code 几乎只用 base64（Read 工具直接返回），URL 来源极少见

---

## 7. System Prompt 中的图片  [ P3 ]

**场景**: Claude Code 的系统提示可能包含图片。

**当前状态**: 代理不检查 `system` 数组中的图片。实际探查发现 Claude Code 的 system prompt 仅为文本块，不含图。

**风险**: 极低。但按 Anthropic 协议规范，system 确实可以包含 image block。

---

## 8. 对话历史膨胀  [ P3 ]

**场景**: 长篇对话 + 多张图片 → 每轮请求体不断膨胀 → 接近 10MB 限制。

**影响**: 达到限制后新请求被 413 拒绝。

**应对**:
- 当前 10MB 限制已生效
- Claude Code 本身有 context compaction 机制，会自动压缩历史，代理只需保持限制即可
- 暂不需要主动压缩历史

---

## 9. 并发与限流  [ P3 ]

**场景**: 多个用户同时用代理 → vision API 频率限制（429）。

**当前状态**: `recognize()` 已实现单次 429 重试（等 1s × 1 次）。并发时可能连续 429。

**待完善**:
- 多个 429 → 指数退避
- 全局并发控制（限制同时 vision 请求数）

---

## 决策链路 — 轻量意图识别模型

### 设计目标: 逼近原生多模态能力

```
原生多模态模型                    当前代理 (外挂识图器)         目标 (加决策模型后)
─────────────────              ────────────────────         ─────────────────────
图片 → token → 注意力         图片 → 文本描述 → 模型         图片 → 文本(缓存) ← 决策模型
用户提问 → 注意力 → 图片        用户提问 → 模型只能看文本      用户提问 → 决策模型 → 定位图片
任意时刻关联任意图              图仅首次识别, 后续无关联       主动重识/对比/聚焦, 全自动
多图天然可比较                  多图仅首次分别描述            决策模型触发对比重识
```

决策模型不是在"查漏补缺"——它承担的是**原生多模态模型中注意力机制的角色**。
文本模型 + 决策模型 + 视觉模型, 三者协同逼近单一体多模态模型的能力。

### 架构

```
请求进入
    │
    ▼
┌─────────────────┐     latest_only: 图在最新消息?
│  Stage 1-2      │──── 否 ──→ 纯文本直通
│  格式检测+图片提取 │
└────────┬────────┘
         │ 是, 提取最新图
         ▼
┌─────────────────────────────────────┐
│        轻量决策模型 (意图识别)         │
│                                      │
│  输入 (固定大小, <500 token, <0.5s):  │
│    - 用户最新消息 (1 句)              │
│    - 图片缓存摘要 (每图 1 行)          │
│    - 上一轮对话最后 1-2 句            │
│                                      │
│  输出:                               │
│    {                                 │
│      action: "re_vision"|"skip",    │
│      image_hash: "abc123...",       │
│      focus_prompt: "...",           │
│      reasoning: "..."               │
│    }                                 │
└──────────┬──────────────────────────┘
           │
     ┌─────┴─────┐
     ▼           ▼
  re_vision    skip
     │           │
     │           └──→ 用缓存描述, 直通
     │
     ▼
┌─────────────────────┐
│  Vision (qwen3.7-plus) │
│  prompt +=          │
│  "用户追问: {focus}" │
│  结果更新缓存         │
└────────┬────────────┘
         │
         ▼
      Rewrite → Forward
```

### 三条路径

```
┌──────────────────────────────────────────────────────────────────────┐
│                                                                      │
│  PATH A: 新图首次                                                     │
│  ─────────────────                                                   │
│  用户: "描述这张图"  +  base64 图片                                    │
│    │                                                                 │
│    ├─ latest_only: 图在最新消息 ✓                                     │
│    ├─ 决策模型: 缓存无此图 → action=re_vision, focus=通用描述           │
│    ├─ vision: 1 次 (8-12s)                                           │
│    └─ 缓存: img_hash → desc                                          │
│                                                                      │
│  PATH B: 追问细节 (图在历史, 用户主动重读)                               │
│  ───────────────────────────────────────                              │
│  用户: "打开 image_123.png 看右上角写的什么"  +  新 base64              │
│    │                                                                 │
│    ├─ latest_only: 图在最新消息 ✓                                     │
│    ├─ 决策模型: 用户关注 "右上角文字" → action=re_vision, focus=右上角   │
│    ├─ vision: 1 次 (带上 focus prompt)                                │
│    └─ 缓存更新: (img_hash, focus) → 新 desc                           │
│                                                                      │
│  PATH C: 无关追问                                                     │
│  ─────────────                                                       │
│  用户: "今天是几号" (图在历史中)                                       │
│    │                                                                 │
│    ├─ latest_only: 图不在最新消息 ✗                                   │
│    └─ → 纯文本直通, 0ms                                               │
│                                                                      │
│  PATH D: 语义追问但未重读 (决策模型主动提取)                             │
│  ─────────────────────────────────────                                │
│  用户: "之前那张甘特图谁负责颜色审批" (图在历史, 不在最新消息)             │
│    │                                                                 │
│    ├─ 决策模型分析:                                                    │
│    │   输入: 用户最新消息 + 缓存摘要 + 历史中所有图的哈希列表              │
│    │   输出: {                                                        │
│    │     action: "re_vision",                                        │
│    │     image_hash: "abc123...",  ← 指定要重读哪张图                   │
│    │     focus_prompt: "重点关注颜色审批相关的信息"                      │
│    │   }                                                             │
│    │                                                                 │
│    ├─ 代理根据 image_hash 从完整请求体中定位对应图片                     │
│    │   (遍历所有 messages, 不限于 latest)                              │
│    │                                                                 │
│    ├─ Vision: 识图 (带上 focus_prompt)                                │
│    │                                                                 │
│    ├─ 更新缓存 → Rewrite → Forward                                    │
│    │                                                                 │
│    └─ 全程用户无感知, 不需要 "提示用户重读"                              │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

### 决策模型输入/输出定义

```
┌─── 输入 ──────────────────────────────────────────────┐
│                                                        │
│  {                                                     │
│    "last_user_message": "右上角那个标签写的什么?",       │
│    "cached_images": [                                  │
│      {                                                  │
│        "id": 1,                                        │
│        "hash": "abc123",                               │
│        "summary": "项目管理甘特图, 深色主题,             │
│                    任务状态, 4/27~5/9 时间线..."         │
│      },                                                │
│      {                                                  │
│        "id": 2,                                        │
│        "hash": "def456",                               │
│        "summary": "微服务架构图, 4层架构,                │
│                    API网关+服务+DB..."                   │
│      }                                                │
│    ],                                                  │
│    "last_assistant_reply": "这是一张甘特图截图...",      │
│    "last_user_message_before": "有哪些颜色"              │
│  }                                                     │
│                                                        │
├─── 输出 ───────────────────────────────────────────────┤
│                                                        │
│  {                                                     │
│    "action": "re_vision" | "skip",                     │
│    "image_hash": "abc123...",  // 目标图片哈希           │
│    "focus_prompt": "请仔细查看图中右上角的标签文字内容",  │
│    "reasoning": "用户追问图中特定位置的文字,              │
│                  缓存摘要未覆盖此细节"                   │
│  }                                                     │
│                                                        │
│  代理收到 action=re_vision 后:                          │
│    1. 遍历请求体所有 messages 找到匹配哈希的图片           │

│    3. 更新缓存 → Rewrite → Forward                       │
│                                                        │
└────────────────────────────────────────────────────────┘
```

### 多图场景 — 决策模型核心价值

```
用户: "对比之前那两张架构图，哪个更合适微服务场景"
  (图1 和 图2 都在历史消息中, 不在最新消息)

    │
    ├─ 决策模型分析:
    │   输入: 用户最新消息 + 缓存摘要列表 + 全部图哈希
    │         图1: 摘要 "微服务架构图A, 4层..."
    │         图2: 摘要 "微服务架构图B, 3层..."
    │
    │   输出: {
    │     action: "re_vision",
    │     image_hashes: ["abc123", "def456"],  ← 同时重识两张
    │     focus_prompt: "请对比两张架构图, 从微服务适用性角度分析各自的优劣"
    │     mode: "compare"                       ← 对比模式
    │   }
    │
    ├─ 代理遍历请求体, 找到两张图
    │
    ├─ Vision 并行识别 (两张图 + 对比 prompt)
    │
    ├─ 重写: [图片对比分析]\n图1: 4层架构...\n图2: 3层架构...\n对比结论: ...
    │
    └─ Forward → 目标模型收到完整对比描述

单图追问 vs 多图对比 — 决策模型统一处理, prompt 自动适配。
```

### 各方案对比

```
┌─────────────┬──────────────┬──────────────┬──────────────────────┐
│             │   关键词规则   │   决策模型     │  始终重识 (最简)      │
├─────────────┼──────────────┼──────────────┼──────────────────────┤
│ 意图覆盖     │  低 (规则有限) │  高 (语义理解) │  100% (不判断)       │
│ 决策速度     │  <1ms        │  <500ms       │  0ms (不决策)        │
│ 识别成本     │  0            │  ~0.01￥       │  0                   │
│ 实现复杂度   │  低           │  中            │  极低                 │
│ 多图定位     │  不支持        │  支持          │  不适用              │
│ 演进能力     │  加规则即可    │  模型泛化      │  不适用              │
│ 总延迟       │  规则开销 0   │  +决策 0.5s    │  每次都 vision 8-12s │
└─────────────┴──────────────┴──────────────┴──────────────────────┘
```

| 优先级 | 边界情况 | 当前状态 | 建议 |
|--------|----------|----------|------|
| P0 | 缓存不全面 | 已识别，有方案 | 关键词触发重识 |
| P1 | 多张图片 | 部分支持，需调格式 | 多图编号 |
| P1 | Vision 不可用 | 已降级，无感知 | 加监控+用户提示 |
| P2 | 超大图 | Content-Length 已拦截 | 单图大小限制 |
| P2 | 流式等待 | 无进度通知 | SSE 状态事件 |
| P3 | 图片 URL | 基础支持 | URL 缓存策略 |
| P3 | System 图片 | 未处理 | 低优，实测不存在 |
| P3 | 历史膨胀 | 已有限制 | 暂不处理 |
| P3 | 并发限流 | 基础重试 | 指数退避 |
