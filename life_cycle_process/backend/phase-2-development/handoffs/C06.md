## middleware/auth.py 设计要点

### APIKeyMiddleware
- Starlette `BaseHTTPMiddleware`，通过 `app.add_middleware()` 注入
- 构造参数 `api_key: str` — 从 `ProxyConfig.proxy_api_key` 传入
- 公开路径 `["/", "/health"]` 始终跳过认证
- 空 `api_key` → 宽松模式（所有请求放行）
- `x-api-key` header 缺失或不匹配 → 返回 `401 {"error": "invalid_api_key"}`

```
请求到达 → 公开路径? → 放行
        → api_key 为空? → 放行
        → x-api-key 匹配? → 放行
        → 否则 → 401
```

## pipeline.py 设计要点

### 两个入口函数

| 函数 | 用途 | 可测试性 |
|------|------|---------|
| `run_pipeline(request, config, **kwargs)` | HTTP 入口 — Content-Length 检查 + JSON 解析 → 委托 `process_request` | 通过 TestClient |
| `process_request(body, path, config, **kwargs)` | 核心管道逻辑 — 接收 dict + path，无需 HTTP Request | `pytest.mark.asyncio` 直接调用 |

### 可注入回调（用于测试和后续任务接入）

| 参数 | 对应阶段 | 当前状态 |
|------|---------|---------|
| `vision_recognize` | Stage 3: Vision Client | 可注入；未注入时跳过 |
| `rewrite_request` | Stage 4: Request Rewriter | 可注入；未注入时使用 original_body |
| `forward_to_target` | Stage 6: Target Client | 可注入；未注入时返回 placeholder response |

### 管道分支逻辑

```
process_request(body, path, config)
  │
  ├─ messages 为空? → HTTP 400
  │
  ├─ Stage 1: Format Detector (detect_format + parse)
  │
  ├─ Stage 2: Image Check (has_images)
  │     │
  │     ├─ 无图片 → 纯文本快速路径
  │     │         → body_to_forward = original_body
  │     │         → forward_to_target(original_body) 或 placeholder
  │     │
  │     └─ 有图片 → 完整管道
  │              → extract_images()
  │              → vision_recognize(images)  [可注入]
  │              → rewrite_request(...)       [可注入]
  │              → body_to_forward = rewritten_body
  │              → forward_to_target(...) 或 placeholder
```

### 业务规则

| 规则 | 位置 | 违反行为 |
|------|------|---------|
| Content-Length < 10MB | `run_pipeline` (读 body 前检查 header) | 413 `{"error": "payload_too_large"}` |
| messages 非空 | `process_request` | 400 `{"error": "messages_required"}` |
| JSON 解析失败 | `run_pipeline` | 400 `{"error": "invalid_request"}` |

## 注意事项

1. **管道注入点设计**：`vision_recognize`、`rewrite_request`、`forward_to_target` 三个可注入回调设计允许 C02/C03/C05 任务直接接入，无需修改管道代码。未注入时管道使用安全默认行为（跳过对应阶段）。
2. **Content-Length 检查在 body 读取前**：通过读取 `request.headers.get("content-length")` 实现，避免先消费 body 再判断大小的问题。
3. **middleware 无配置依赖**：`APIKeyMiddleware` 的 `api_key` 通过构造参数传入，不直接依赖 `ProxyConfig`，便于单元测试。
4. **routes/ 模块保留**：`routes/anthropic.py` 和 `routes/openai.py` 文件未被删除（保持向后兼容），但 `app.py` 不再调用它们。管道逻辑已统一收敛到 `pipeline.py`。
5. **placeholder forward 返回结构化 JSON**：`{"status": "forwarded", "source_format": ..., "target_model": ..., ...}` — 此格式被后续 C05 真实 Target Client 替换后不再返回，仅用于早期测试验证管道走通。
